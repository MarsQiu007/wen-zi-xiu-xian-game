# 游戏逻辑重探索与优化设计方案

## TL;DR
> **Summary**: 以不重写核心框架为前提，重构“开局语义、日志可读性、角色可观测性、地图可感知性”四条主线，确保 New Game 每次生成全新世界、Continue 才恢复旧存档，并显著提升事件叙事密度与玩家理解成本。  
> **Deliverables**:
> - 主菜单（New Game / Continue / Mode）与启动路由
> - 存档快照闭环（保存+恢复+版本）
> - 事件去重与多层日志展示
> - 角色名册/详情/经历时间线
> - 文本优先地图与角色位置联动
> **Effort**: Large  
> **Parallel**: YES - 3 waves  
> **Critical Path**: 启动语义锁定 → 快照恢复闭环 → 日志治理 → 角色与地图可视化

## Context
### Original Request
用户反馈：日志过少且“秋收祭”重复导致看不懂世界演化；无法查看其他角色经历与属性；没有地图概念看不出主要角色位置；无设置/主菜单无法选择新游戏、继续、模式；且 New Game 必须每次全新初始化世界（NPC/地图全新）。

### Interview Summary
- 目标不是“修补 UI 文案”，而是完整重探：入口语义、存档语义、日志系统、角色观测、地图能力。
- 必须维持 Godot 4.6 风格：Scene + Script + Resource + signal，避免大规模框架重写。
- 本次输出为执行计划，不直接改业务代码。

### Metis Review (gaps addressed)
- 已补强的关键缺口：
  - 明确 New/Continue 语义必须分离且可自动验收。
  - 明确“玩家随机 seed 与测试固定 seed”双轨策略。
  - 明确日志治理要有量化阈值（重复率、窗口冷却、模板覆盖率）。
  - 明确地图一期仅做文本优先（区域+位置），延期复杂寻路和图形地图。
  - 明确角色可见性按 `human_visible/deity_visible` 做权限边界。

### Decision Locks (implementation defaults)
- 存档策略（一期）：**单槽位稳定优先**，路径 `user://save/slot_1.json`；保留扩展多槽结构但不在一期开放 UI。
- New/Continue 语义：
  - **New Game**：总是新 seed + 全量 reset，不读取旧快照。
  - **Continue**：只走 `load_snapshot`，恢复失败则回菜单并提示。
- Seed 策略：
  - 玩家正常开局：随机 seed（非 42）。
  - 自动化测试：显式注入固定 seed。
- 地图可见性（一期）：**全知视角**（显示主要角色当前位置），后续再做可见性战争迷雾。
- 角色可见字段（一期）：基础属性 + 运行时核心状态 + 最近时间线；隐藏字段继续受 `human_visible/deity_visible` 过滤。
- 日志阈值（一期）：
  - 单模板占比上限 35%（200 tick）
  - 同 key 10 tick 窗口重复上限 2
  - 标准视图默认显示 actor + location + cause/result。

## Work Objectives
### Core Objective
在不推翻现有模拟核心的前提下，交付“可理解、可追溯、可选择、可定位”的首版体验：玩家能明确区分 New 与 Continue，能读懂世界发生了什么，能查看关键角色经历与属性，能在地图层面知道角色所在区域。

### Deliverables
- 启动路由：主菜单 + 模式选择 + New/Continue 分支
- 存档闭环：save slot/version/atomic write 与 runtime snapshot restore
- 日志升级：去重冷却 + 多层摘要 + 过滤/按角色视图
- 角色视图：名册、属性详情、经历时间线
- 地图视图：区域层级/邻接 + 角色位置高亮 + 位置变更事件

### Definition of Done (verifiable)
- [ ] New Game 连续两次启动得到不同 `world_fingerprint`（默认随机）。
- [ ] Continue 可恢复保存前核心状态（time/seed/mode/actor_region/log_cursor）。
- [ ] 200 tick 测试中同类事件重复率低于阈值（计划默认 35%）。
- [ ] 至少 1 名非主角可查看“属性 + 最近事件时间线”。
- [ ] 地图面板可显示主要角色当前位置并在移动后刷新。

### Must Have
- New/Continue 语义严格分离。
- 日志必须显示 `actor + location + cause/result` 基础叙事三要素。
- 角色面板必须支持“其他角色”可查看（按可见性规则过滤）。
- 地图必须具备“区域感 + 角色位置感”。

### Must NOT Have
- 不重写 SimulationRunner 核心架构。
- 不引入复杂图形地图/寻路系统（一期）。
- 不扩展到多世界切换、云存档、跨版本自动迁移全套系统（一期）。
- 不把大量业务塞进单个 Autoload（仅聚合查询/状态协调）。

## Verification Strategy
> ZERO HUMAN INTERVENTION - all verification is agent-executed.

- Test decision: tests-after（Godot headless + runtime script validation）
- QA policy: 每个任务都含 happy/failure 场景
- Evidence: `.sisyphus/evidence/task-{N}-{slug}.txt|json|png`

### Hard Acceptance Gates (AC-01 ~ AC-08)
- AC-01 New Game 两次 `world_fingerprint` 不同（默认随机）。
- AC-02 Continue 恢复后 `world_fingerprint` 与保存时一致。
- AC-03 Release 流程 New Game 默认 seed 非固定 42。
- AC-04 启动先进入主菜单；未选择前不进入 simulation。
- AC-05 模式选择可保存并在 Continue 后保持一致。
- AC-06 日志防刷生效（冷却窗口内重复折叠/抑制）。
- AC-07 角色查询返回属性快照 + 最近时间线（非空且可排序）。
- AC-08 地图返回主要角色有效区域并在位置变更后刷新。
